# Объяснение путей и команд Linux

## Структура файловой системы Linux

Linux использует единую иерархическую файловую систему, начинающуюся с корневого каталога `/`

### Основные системные каталоги:

```
/                    # Корневой каталог (root)
├── etc/             # Конфигурационные файлы системы
├── var/             # Переменные данные (логи, кэш)
├── usr/             # Пользовательские программы
├── home/            # Домашние каталоги пользователей
├── bin/             # Основные системные команды
├── sbin/            # Системные команды для администрирования
└── tmp/             # Временные файлы
```

## Подробно о /etc

### Что такое /etc
**`/etc`** - сокращение от "et cetera" (и так далее), содержит конфигурационные файлы системы.

### Характеристики /etc:
- **Только конфигурации**: никаких исполняемых файлов или данных
- **Общесистемные настройки**: влияют на всех пользователей
- **Редактирование через sudo**: обычные пользователи не могут изменять
- **Текстовые файлы**: большинство конфигов в читаемом формате

### Примеры важных файлов в /etc:
```bash
/etc/passwd          # Информация о пользователях
/etc/hosts           # Локальные DNS записи
/etc/nginx/          # Конфигурации nginx
/etc/apache2/        # Конфигурации Apache
/etc/ssh/            # Настройки SSH
/etc/fstab           # Настройки монтирования дисков
```

## Структура конфигураций nginx

### /etc/nginx/nginx.conf
**Главный конфигурационный файл nginx** - содержит глобальные настройки:
- Количество рабочих процессов
- Настройки логирования
- Включение дополнительных конфигов

### /etc/nginx/sites-available/
**Каталог доступных сайтов** - здесь хранятся конфигурации всех сайтов:
- Каждый сайт = отдельный файл
- Файлы НЕ активны автоматически
- Можно создавать и редактировать без влияния на работу

### /etc/nginx/sites-enabled/
**Каталог активных сайтов** - содержит символические ссылки на файлы из sites-available:
- Только ссылки, не реальные файлы
- nginx читает только отсюда
- Активация/деактивация через создание/удаление ссылок

## Команды и их назначение

### sudo mkdir -p /etc/nginx/ssl

```bash
sudo                 # Выполнить от имени администратора
mkdir                # Создать каталог
-p                   # Создать промежуточные каталоги если нужно
/etc/nginx/ssl       # Путь для создания
```

**Зачем /etc/nginx/ssl:**
- Стандартное место для SSL сертификатов nginx
- Защищено правами доступа (только root может читать)
- Логически организовано (ssl файлы рядом с конфигами nginx)

### sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048

```bash
sudo                 # Права администратора (для записи в /etc)
openssl              # Утилита для работы с криптографией
req                  # Создать запрос сертификата
-x509                # Создать самоподписанный сертификат
-nodes               # Не шифровать приватный ключ паролем
-days 365            # Срок действия сертификата (1 год)
-newkey rsa:2048     # Создать новый RSA ключ размером 2048 бит
-keyout              # Куда сохранить приватный ключ
-out                 # Куда сохранить сертификат
```

### sudo nano /etc/nginx/sites-available/web_labs

```bash
sudo                           # Права администратора
nano                           # Текстовый редактор
/etc/nginx/sites-available/   # Каталог для конфигов сайтов
web_labs                      # Имя файла конфигурации
```

**Почему sites-available:**
- Стандарт Debian/Ubuntu для организации конфигов
- Разделение доступных и активных конфигураций
- Безопасное редактирование без влияния на работу

### sudo ln -s /etc/nginx/sites-available/web_labs /etc/nginx/sites-enabled/

```bash
sudo                          # Права администратора
ln                            # Создать ссылку
-s                            # Символическая ссылка (не копия)
/etc/nginx/sites-available/web_labs  # Источник (реальный файл)
/etc/nginx/sites-enabled/     # Назначение (папка активных конфигов)
```

**Принцип работы символических ссылок:**
- Ссылка указывает на другой файл
- Изменения в источнике сразу видны через ссылку
- Удаление ссылки не удаляет исходный файл
- nginx читает только из sites-enabled

### sudo nginx -t

```bash
sudo                 # Иногда нужны права для чтения конфигов
nginx                # Программа веб-сервера
-t                   # Тестировать конфигурацию (не запускать)
```

**Что проверяется:**
- Синтаксис всех конфигурационных файлов
- Доступность указанных файлов (сертификаты, логи)
- Корректность директив и их значений

### sudo systemctl

```bash
sudo systemctl start nginx     # Запустить службу
sudo systemctl stop nginx      # Остановить службу  
sudo systemctl restart nginx   # Перезапустить службу
sudo systemctl reload nginx    # Перезагрузить конфигурацию
sudo systemctl enable nginx    # Включить автозапуск
sudo systemctl status nginx    # Показать статус службы
```

**systemctl** - менеджер служб systemd:
- Управляет фоновыми процессами (демонами)
- Контролирует запуск/остановку служб
- Обеспечивает автозапуск при загрузке системы

## Права доступа и безопасность

### Почему нужен sudo
- Системные каталоги (/etc) защищены от обычных пользователей
- SSL ключи должны быть доступны только администратору
- Управление службами требует административных прав

### Структура прав в /etc/nginx/ssl:
```bash
drwxr-xr-x root root /etc/nginx/ssl/           # Каталог
-rw------- root root selfsigned.key           # Приватный ключ (только root)
-rw-r--r-- root root selfsigned.crt           # Сертификат (можно читать всем)
```

## Логика организации

### Преимущества такой структуры:
1. **Централизация**: все конфиги в одном месте (/etc)
2. **Безопасность**: защита через права доступа
3. **Гибкость**: sites-available/sites-enabled позволяет легко управлять сайтами
4. **Стандартизация**: одинаковая структура на всех Linux системах
5. **Обслуживание**: легко находить и редактировать настройки
